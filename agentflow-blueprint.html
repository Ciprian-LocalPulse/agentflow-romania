BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

  BEFORE UPDATE ON leads
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

Securitate — Row Level Security

Cel mai critic punct al intregii platforme. Regulile RLS din Supabase garanteaza izolarea completa a datelor intre agenti, independent de logica din aplicatie.

⚠
Principiu fundamental: Chiar daca un atacator exploateaza un bug in codul aplicatiei si trimite un request cu un alt user_id, baza de date refuza automat cererea prin politicile RLS. Securitatea nu depinde exclusiv de cod — baza de date este ultimul zid.
TABEL leads RLS ENABLED — FORCE ROW SECURITY
Politica
Definitie SQL
SELECT (citire)
USING (auth.uid() = user_id)
INSERT (creare)
WITH CHECK (auth.uid() = user_id)
UPDATE (editare)
USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id)
DELETE (stergere)
USING (auth.uid() = user_id)
TABEL properties
SELECT
USING (auth.uid() = user_id)
INSERT
WITH CHECK (auth.uid() = user_id)
UPDATE
USING (auth.uid() = user_id)
DELETE
USING (auth.uid() = user_id)
TABEL viewings + followups
SELECT
USING (auth.uid() = user_id)
INSERT
WITH CHECK (auth.uid() = user_id)
UPDATE
USING (auth.uid() = user_id)
DELETE

SPECIAL users — politici speciale
SELECT propriu
USING (auth.uid() = id) — un agent vede DOAR profilul sau
UPDATE propriu
USING (auth.uid() = id) — poate edita DOAR profilul sau
INSERT (admin only)

UPDATE plan (admin)
USING (auth.role() = 'service_role') — planul se actualizeaza DOAR prin Stripe Webhook, niciodata din client
SQL — Activare RLS si creare politici (exemplu complet pentru tabelul leads) 
-- Pasul 1: Activam RLS pe tabel
ALTER TABLE leads ENABLE ROW LEVEL SECURITY;
ALTER TABLE leads FORCE ROW LEVEL SECURITY; -- si pentru owner-ul tabelului

-- Pasul 2: Cream politicile pe fiecare operatie CRUD

CREATE POLICY "Agent poate vedea doar leadurile sale"
  ON leads FOR SELECT
  USING (auth.uid() = user_id);

CREATE POLICY "Agent poate crea leaduri proprii"
  ON leads FOR INSERT
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Agent poate edita leaduri proprii"
  ON leads FOR UPDATE
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Agent poate sterge leaduri proprii"
  ON leads FOR DELETE
  USING (auth.uid() = user_id);

-- Pasul 3: Plan check — limiteaza numarul de leaduri per plan
CREATE OR REPLACE FUNCTION check_lead_limit() RETURNS TRIGGER AS $$
DECLARE
  lead_count INTEGER;
  user_plan TEXT;
  max_leads INTEGER;
BEGIN
  SELECT subscription_plan INTO user_plan FROM users WHERE id = NEW.user_id;
  max_leads := CASE user_plan
    WHEN 'free'    THEN 10
    WHEN 'solo'    THEN 50
    WHEN 'pro'     THEN 9999
    WHEN 'agency'  THEN 9999
    ELSE 10
  END;
  SELECT COUNT(*) INTO lead_count FROM leads WHERE user_id = NEW.user_id;
  IF lead_count >= max_leads THEN
    RAISE EXCEPTION 'Limita de leaduri atinsa pentru planul %', user_plan;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER enforce_lead_limit BEFORE INSERT ON leads
  FOR EACH ROW EXECUTE FUNCTION check_lead_limit();

User Journey — Vizitator → Client Platitor

Traseul complet al unui agent imobiliar de la primul contact cu platforma pana la utilizarea zilnica a contului activ.

1
Descoperire — Landing Page (agentflow.ro)

Vizitatorul intra pe landing page prin reclama Facebook/Google, recomandare sau cautare organica. Vede hero-ul premium, beneficiile, testimonialele si sectiunea de preturi.

Actiuni disponibile pe landing page
CTA Principal: "Incepe gratuit" → /register
CTA Secundar: "Cumpara Plan Solo" → /register?plan=solo
Demo Live: "Vezi demo-ul" → /demo (read-only)
2
Inregistrare — Creare Cont cu Supabase Auth

Pagina /register cu formular simplu: Nume complet, Email, Parola. Optinal: Google OAuth si Facebook OAuth prin Supabase Social Providers. La submit, Supabase Auth creeaza contul si trimite email de confirmare.

Date colectate la register
  • Nume complet
  • Email (unic in sistem)
  • Parola (hash bcrypt — niciodata in plain text)
  • Plan ales (din query param ?plan=solo)
Flow Supabase Auth
  • supabase.auth.signUp() emite JWT
  • Email confirmare trimis automat
  • Trigger DB creeaza randul in users
  • Redirect catre /onboarding sau /pay
3
Verificare Abonament — Middleware Auth

La fiecare acces in aplicatie, middleware-ul Next.js verifica simultan: (1) exista JWT valid din Supabase Auth? (2) subscription_status = 'active' in baza de date? Ambele conditii trebuie indeplinite.

DA — are abonament activ

Redirect catre /dashboard. Acces complet la toate modulele conform planului. JWT-ul Supabase este valid 1 ora, refreshed automat.

NU — fara abonament sau expirat

Redirect catre /subscribe cu planul recomandat precompletat. Daca planul a expirat: banner de warning, acces read-only 7 zile grace period.

4
Plata — Checkout Stripe (daca NU are abonament)

Pagina /subscribe afiseaza cele 3 planuri cu buton "Cumpara". Click → API nostru creeaza Stripe Checkout Session → redirect catre pagina Stripe hosteduite secure (PCI-DSS compliant).

Flow plata Stripe
  • POST /api/stripe/create-checkout-session
  • Server creeaza stripe.checkout.sessions.create()
  • Client primeste session.url si este redirectat
  • Clientul introduce cardul pe pagina Stripe
  • Stripe trimite webhook la /api/stripe/webhook
  • Webhook actualizeaza subscription_plan in DB
  • Redirect automat catre /dashboard?welcome=1
Webhook events tratate
  • checkout.session.completed → activare plan
  • customer.subscription.updated → schimbare plan
  • customer.subscription.deleted → dezactivare
  • invoice.payment_failed → email reminder + grace
  • invoice.payment_succeeded → reactivare daca past_due
5
Dashboard — Utilizare Zilnica

Agentul are acces complet la toate modulele. Fiecare actiune (adauga lead, creeaza proprietate etc.) trimite request autentificat la API, care valideaza JWT si executa operatia cu user_id din token — niciodata din body-ul requestului.

Principiu critic de securitate
// GRESIT — user_id din body (vulnerabil la manipulare)
const { user_id, lead_data } = req.body; // NICIODATA asa

// CORECT — user_id EXCLUSIV din JWT verificat server-side
const { data: { user } } = await supabase.auth.getUser(jwt_token);
const user_id = user.id; // Sigur — nu poate fi falsificat
await supabase.from('leads').insert({ ...lead_data, user_id });

TypeScript — Middleware Next.js (src/middleware.ts) — protectie rute 
import { createMiddlewareClient } from '@supabase/auth-helpers-nextjs';
import { NextRequest, NextResponse } from 'next/server';

export async function middleware(req: NextRequest) {
  const res = NextResponse.next();
  const supabase = createMiddlewareClient({ req, res });

  // Verifica sesiunea activa
  const { data: { session } } = await supabase.auth.getSession();

  const isProtectedRoute = req.nextUrl.pathname.startsWith('/dashboard')
    || req.nextUrl.pathname.startsWith('/api/leads')
    || req.nextUrl.pathname.startsWith('/api/properties');

  if (isProtectedRoute && !session) {
    return NextResponse.redirect(new URL('/login?reason=unauthenticated', req.url));
  }

  // Verifica plan activ pentru rute premium
  if (session) {
    const { data: profile } = await supabase
      .from('users').select('subscription_status,subscription_plan')
      .eq('id', session.user.id).single();

    if (isProtectedRoute && profile?.subscription_status !== 'active') {
      return NextResponse.redirect(new URL('/subscribe?reason=inactive', req.url));
    }
  }
  return res;
}

export const config = { matcher: ['/dashboard/:path*', '/api/(leads|properties|viewings|followups|stripe/create-:path*)'] };

AI Engine — Generare Descrieri OpenAI

Arhitectura completa a modulului AI: de la click-ul agentului pana la descrierea stocata in baza de date.

Flux complet generare AI
1
Agentul apasa "Genereaza descriere AI" pe o proprietate. Alege tonul dorit (Premium / Emotional / Investitional / Minimalist).
2
Frontend trimite POST /api/properties/[id]/generate-ai cu payload: { tone: 'premium' }. JWT-ul este inclus automat in Authorization header.
3
Server verifica: (1) JWT valid, (2) proprietatea apartine agentului, (3) plan Pro/Agency, (4) sub limita de generari/zi via Redis counter.
4
Server construieste prompt-ul cu datele proprietatii si apeleaza openai.chat.completions.create() cu modelul gpt-4o.
5
Descrierea returnata de OpenAI este salvata in campul descriere_ai din tabelul properties, asociata cu user_id si property_id.
✓
Frontend primeste descrierea si o afiseaza in editor. Agentul o editeaza dupa nevoie si o copiaza direct in anuntul de pe imobiliare.ro sau OLX.
TypeScript — /api/properties/[id]/generate-ai/route.ts 
import OpenAI from 'openai';
import { Ratelimit } from '@upstash/ratelimit';
import { Redis } from '@upstash/redis';

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });
const ratelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(5, '1 d'), // 5/zi plan Pro
});

export async function POST(req: Request, { params }) {
  const user = await getAuthUser(req); // verifica JWT
  if (!user) return unauthorized();

  const { success } = await ratelimit.limit(user.id);
  if (!success) return tooManyRequests('Limita AI atinsa pentru azi');

  const property = await getPropertyOwned(params.id, user.id);
  if (!property) return notFound();

  const { tone } = await req.json();
  const prompt = buildPropertyPrompt(property, tone);

  const completion = await openai.chat.completions.create({
    model: 'gpt-4o',
    temperature: 0.7,
    max_tokens: 600,
    messages: [
      { role: 'system', content: 'Esti un expert imobiliar roman care scrie anunturi profesionale in romana.' },
      { role: 'user', content: prompt }
    ]
  });

  const descriere_ai = completion.choices[0].message.content;

  await supabase.from('properties')
    .update({ descriere_ai })
    .eq('id', params.id).eq('user_id', user.id);

  return json({ descriere_ai });
}

TypeScript — /api/stripe/webhook/route.ts — receptor complet 
import Stripe from 'stripe';
const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!);

export async function POST(req: Request) {
  const body = await req.text();
  const sig = req.headers.get('stripe-signature')!;

  // CRIRIC: Verifica semnatura Stripe — respinge orice request fals
  let event: Stripe.Event;
  try {
    event = stripe.webhooks.constructEvent(body, sig,
      process.env.STRIPE_WEBHOOK_SECRET!);
  } catch {
    return json({ error: 'Invalid signature' }, { status: 400 });
  }

  switch (event.type) {
    case 'checkout.session.completed': {
      const session = event.data.object;
      await activateSubscription({
        stripeCustomerId: session.customer,
        subscriptionId: session.subscription,
        plan: getPlanFromSession(session),
      });
      break;
    }
    case 'customer.subscription.deleted': {
      await deactivateSubscription(event.data.object.customer);
      // Trimite email via Resend: "Abonamentul tau a expirat"
      break;
    }
    case 'invoice.payment_failed': {
      await setSubscriptionPastDue(event.data.object.customer);
      await sendPaymentFailedEmail(event.data.object.customer_email);
      break;
    }
    case 'customer.subscription.updated': {
      // Upgrade/downgrade plan — actualizeaza subscription_plan in DB
      await updatePlan(event.data.object);
      break;
    }
  }
  return json({ received: true });
}

async function activateSubscription({ stripeCustomerId, subscriptionId, plan }) {
  // Actualizeaza DB — EXCLUSIV prin service_role, nu prin utilizator
  await supabaseAdmin.from('users').update({
    subscription_plan: plan,
    subscription_status: 'active',
    stripe_subscription_id: subscriptionId,
  }).eq('stripe_customer_id', stripeCustomerId);
}

Autentificare — Supabase Auth Flow

Mecanismul complet de autentificare: JWT lifecycle, refresh tokens, OAuth social providers si protectia sesiunilor.

JWT Token Lifecycle
  • La login: Supabase emite access_token (JWT, 1 ora) + refresh_token (7 zile)
  • JWT contine: user.id, email, role, expirat la (exp)
  • La fiecare request: middleware valideaza JWT cu cheia publica Supabase
  • La 55 min: Supabase JS client auto-refresheaza token via refresh_token
  • La logout: refresh_token invalidat in Supabase → toate sesiunile expire
  • Token-urile sunt stocate in cookies httpOnly, SameSite=Lax, Secure
Social Login — OAuth Providers

La primul login OAuth, Supabase creeaza automat contul si declanseaza triggerul care insereaza randul in tabelul users cu subscription_plan = 'free'.

SQL — Trigger creare profil la inregistrare 
-- Se declanseaza automat cand Supabase Auth creeaza un user nou
CREATE OR REPLACE FUNCTION handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.users (id, email, nume_complet, subscription_plan, subscription_status)
  VALUES (
    NEW.id,
    NEW.email,
    COALESCE(NEW.raw_user_meta_data->>'full_name', 'Agent Nou'),
    'free',
    'inactive'
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- Triggerul se aplica pe tabelul intern Supabase Auth
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION handle_new_user();
Masuri de securitate aditionale
  • Rate limiting pe /api/auth/*: max 10 req/min per IP via Upstash
  • Email confirmare obligatorie inainte de acces in dashboard
  • Brute-force protection: Supabase blocheaza dupa 5 tentative esuate
  • CORS configurat strict: doar origini agentflow.ro permise
  • Headers securitate: CSP, X-Frame-Options, HSTS activate pe Vercel
  • Service role key EXCLUSIV pe server — niciodata expusa clientului
